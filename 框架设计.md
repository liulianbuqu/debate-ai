# 🎯 辩手AI辅助写作系统 — 框架设计

## 一、核心理念

> 不是让AI替你写，而是让AI**学会你的写法**。

本系统的核心机制是一个 **「迭代学习 + 风格画像」** 的双循环：

```
         ┌─────────────────────────────────┐
         │         第1层：内容生成           │
         │  用户输入辩题 → AI生成稿件 → 反馈   │
         └─────────────────────────────────┘
                        ↕
         ┌─────────────────────────────────┐
         │         第2层：风格学习           │
         │  上传素材 → 分析风格 → 更新画像    │
         └─────────────────────────────────┘
```

- **第1层**解决"快速出稿"的需求
- **第2层**解决"越用越懂你"的需求
- 两层互相促进：素材越多 → 画像越准 → 稿件越好 → 越愿意用

### 技术选型

## 四、核心流程设计

### 流程1：上传素材 → 更新风格画像

```
用户上传一篇立论稿
        │
        ▼
后端接收稿子
        │
        ▼
AI分析（自动提取）：
  ├─ 论证结构识别
  ├─ 语言风格标注
  ├─ 核心论点提取
  ├─ 修辞手法统计
  └─ 常用词汇分析
        │
        ▼
合并到现有风格画像
（加权平均，新素材权重更高）
        │
        ▼
返回更新后的画像摘要
```

### 流程2：生成稿件

```
用户输入：
  ├─ 辩题
  ├─ 持方（正/反）
  ├─ 稿子类型
  └─ 额外要求
        │
        ▼
系统组装 Prompt：
  ├─ 当前风格画像（告诉AI这个人的风格）
  ├─ 参考素材（选取最相关的2-3篇）
  ├─ 辩论理解（告诉AI这个人的辩论观）
  └─ 用户本次输入
        │
        ▼
调用 DeepSeek API
        │
        ▼
返回稿件
        │
        ▼
用户反馈 → 记录到生成日志 → 触发画像微调
```

### 流程3：迭代优化循环 ⭐ — 核心创新

```
第1次使用：
  上传2篇立论稿 → 建立基础画像 → 生成新立论稿
  → 可能只有60%满意

第5次使用：
  上传10篇各类稿件 → 画像丰富 → 生成质量明显提升
  → 可能有75%满意

第20次使用：
  积累了丰富的素材库 + 精确的风格画像 + 反馈历史
  → 生成的稿件只需微调即可使用
  → 可能达到85-90%满意
```

## 五、Prompt 工程体系

这是系统能否真正"懂辩论"的关键。我们将设计一套**分层Prompt体系**：

### System Prompt 分层

```
Layer 1：辩论专家身份设定（固定）
  └─ "你是一位精通华语辩论的资深辩手..."
  
Layer 2：用户风格画像（动态更新）
  └─ 从风格画像表动态注入
  
Layer 3：辩论理解框架（动态更新）
  └─ 用户的辩论观、方法论
  
Layer 4：参考素材（按需选取）
  └─ 与当前任务最相关的用户历史稿件
  
Layer 5：当前任务指令（每次输入）
  └─ 辩题、持方、稿子类型、要求
```

### 示例 Prompt 结构（以立论稿为例）

```
【系统身份】
你是一位拥有十年经验的华语辩论资深辩手和教练...

【当前辩手的风格画像】
语言风格：偏向犀利、节奏感强
论证方式：喜欢用定义先行+案例印证
结构偏好：三层递进式
常用修辞：排比、反问
注意：避免过于学术化的表达，要有"人味"

【辩论理解】
这位辩手认为：立论的核心不是下定义，而是建立判断标准...
质询的核心是取得共识，而不是压倒对手...

【参考素材】
（此处插入用户自己的立论稿片段2-3篇）

【本次任务】
辩题：XX
持方：正方/反方
要求：...

【输出规范】
- 字数800-1000字
- 第一段点明标准
- 每段都需要一个"金句"收尾
- ...
```

## 六、数据库表结构

```sql
-- 辩手画像表
CREATE TABLE debater_profile (
    id TEXT PRIMARY KEY,
    name TEXT,
    debate_understanding TEXT,  -- JSON
    style_profile TEXT,         -- JSON
    frequent_vocabulary TEXT,   -- JSON
    update_count INTEGER DEFAULT 0,
    created_at TIMESTAMP,
    last_updated TIMESTAMP
);

-- 素材表
CREATE TABLE materials (
    id TEXT PRIMARY KEY,
    type TEXT NOT NULL,          -- 立论稿/质询稿/陈词稿/自由辩战场稿/辩论理解
    title TEXT NOT NULL,
    content TEXT NOT NULL,
    tags TEXT,                   -- JSON array
    context TEXT,                -- JSON
    extracted_features TEXT,     -- JSON
    created_at TIMESTAMP
);

-- 生成日志表
CREATE TABLE generation_logs (
    id TEXT PRIMARY KEY,
    type TEXT NOT NULL,
    request TEXT NOT NULL,       -- JSON
    response TEXT NOT NULL,      -- JSON
    feedback TEXT,               -- JSON
    created_at TIMESTAMP
);
```

## 九、关键设计决策说明

### 为什么使用 DeepSeek API 而不是其他方案？
- 用户指定，且有较高的性价比优势
- 支持超长上下文（适合注入风格画像+参考素材）
- 中文能力强，适合辩论这种需要语言精妙度的场景

### 为什么风格画像要"加权更新"？
- 新上传的素材更能反映辩手当前的风格
- 但老的经典作品依然有参考价值
- 公式：`新分值 = 旧分值 × (1-α) + 新分析值 × α`
- α随素材总量动态调整，初始设为0.6

### 为什么需要"辩论理解"字段？
- 辩论写作不只是技术活，更是价值观的体现
- 记录用户对辩论本质的理解，让AI不只是模仿"形"，更理解"神"
- 这部分需要在用户使用过程中不断提炼和深化
